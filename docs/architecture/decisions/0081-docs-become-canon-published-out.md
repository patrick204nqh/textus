# ADR 0081 — Docs become canon, published out to `docs/`

**Date:** 2026-06-04
**Status:** Accepted
**Touches:** [ADR 0050](./0050-native-authoring-and-content-identical-adoption.md) (content-identical adoption — the `guard_clobber` branch this migration rides on), [ADR 0046](./0046-publish-leaf-subtrees.md) and [ADR 0047](./0047-publish-tree-keyless-subtree-mirror.md) (the leaf/tree publish modes), [ADR 0052](./0052-typed-publish-block.md) (the typed `publish:` block those modes now fold into), [ADR 0070](./0070-content-addressed-build-artifacts.md) (moved sentinels out of tree to `.run/sentinels/` — why published docs are not annotated), [ADR 0041](./0041-dogfood-textus-in-its-own-repo.md) (textus dogfoods itself — the `knowledge` zone this extends), [ADR 0044](./0044-system-actors-resolved-by-capability.md) (`build` runs as the build-capable actor automatically).

> **One sentence:** textus already dogfoods itself (ADR 0041) with a `knowledge` zone (kind `canon`) that holds only `project.md` + the runbooks, while every other committed doc under `docs/` is an un-owned flat file an agent can silently drift; this ADR makes **every committed `docs/` file canon** — authored under `.textus/zones/knowledge/`, published *back out* to its original `docs/` path by `textus build` — turning `docs/` into a committed, sentinel-managed mirror (the same status `CLAUDE.md`/`AGENTS.md` already hold), using ADR 0050's content-identical adoption so the first `build` is a **zero `git diff`** with links, blame, and layout preserved.

## Context

textus is a durable multi-writer repo-memory tool, and it dogfoods itself (ADR 0041): its own `.textus/` store carries a `knowledge` zone of kind `canon`. Today that zone holds almost nothing — `project.md` and the runbooks — while the bulk of the project's prose (the architecture docs, the how-to/reference/explanation guides, the cookbook, the tutorials, and the **80 ADRs** including this one) lives as ordinary files under `docs/`, outside the store entirely.

That split is the problem. The whole pitch of textus is that durable repo knowledge should be *owned* — entered into a store, attributed, addressable, drift-guarded — rather than left as loose files that any writer (human or agent) can edit, move, or quietly let rot with no record of who or why. `docs/` is precisely the loose-file pile the tool exists to replace, and it is the most-read prose in the repo. A store that doesn't own its own docs is not eating its own dog food.

We already publish prose-heavy, multi-file artifacts in their **native shape** (ADR 0046 whole-leaf subtrees, ADR 0047 key-less tree mirror, now unified under the typed `publish:` block of ADR 0052), and `CLAUDE.md`/`AGENTS.md` are already canon published out of the store. The only thing that made "own `docs/` too" awkward was migration: the files already exist at the exact paths textus would publish to, so a naive first publish trips the clobber guard. ADR 0050 removed that friction — `Ports::Publisher`'s `guard_clobber` adopts an unmanaged target **iff the target's bytes already equal the source** (the adoption branch). That is exactly the state a copy-into-store-then-publish migration produces, which makes a clean, no-diff adoption of `docs/` mechanically available with nothing new to build.

## Decision

1. **Every committed `docs/` file becomes canon.** Each is authored under `.textus/zones/knowledge/` (zone = `knowledge`, kind = `canon`), and `docs/` becomes a committed, sentinel-managed **mirror** regenerated by `textus build` — exactly the status `CLAUDE.md`/`AGENTS.md` already hold. Humans and agents author in the store; `docs/` is output.

2. **Migrate by adopt-in-place (ADR 0050).** For each doc: copy it verbatim into the store and register an entry whose publish target is its *original* `docs/` path. Because the target is byte-identical to the new store source, `guard_clobber`'s adoption branch accepts it — writes the sentinel, leaves the file untouched — so the **first `build` is a zero `git diff` in `docs/`**: no renames, relative links intact, `git blame` history preserved. No delete-then-republish dance, no flag to remember.

3. **Publish modes follow the existing typed `publish:` block (ADR 0052).** Multi-file subtrees (e.g. the ADR log, the guide directories) publish via `publish: { tree: <dir> }` (ADR 0047), mirroring the directory layout 1:1. The two standalone index files — `docs/README.md` and `docs/architecture/README.md` — publish via `publish: { to: [<path>] }` (ADR 0046).

4. **Every doc entry is `schema: null`.** Docs are prose, not structured records. This is load-bearing for the **80 ADRs**, which carry their metadata as markdown (`**Date:**` / `**Status:**` lines), *not* frontmatter. Imposing a schema would require frontmatter on 80 frozen, already-shipped files — pure churn — and would put a load-time validation gate in front of historical records that predate the schema. `schema: null` keeps them adoptable as-is.

5. **Entry keys use hyphens, not underscores.** The key segment grammar is `[a-z0-9][a-z0-9-]*` (`lib/textus/key/grammar.rb:4`) — lowercase, digits, hyphens; underscores are rejected. So directory keys are e.g. `knowledge.how-to`, `knowledge.reference`, `knowledge.architecture.decisions`.

6. **Sentinels stay out of tree (ADR 0070).** Publish sentinels live under `.textus/.run/sentinels/` (gitignored), so the published `docs/` files are **not annotated** with any textus marker — they read as plain, clean docs to anyone who opens them on GitHub.

7. **`build` runs as the build-capable actor automatically (ADR 0044).** No caller-role gymnastics: the existing build-actor resolution covers the docs publish like any other.

8. **Scope excludes `docs/plans/` and `docs/assets/`.** `docs/plans/` is gitignored working scratch (per project convention, plans are local-only and never committed), so there is nothing to own. `docs/assets/` was binary branding, already relocated to repo-root `assets/` in a prior commit; binaries are not canon prose.

## Consequences

- **The process shifts, loudly: humans stop hand-editing `docs/`.** From now on a doc edit goes to the store entry (CLI/MCP `put`, or editing the file under `.textus/zones/knowledge/`), and `build` re-publishes the mirror. Editing `docs/` directly is now editing generated output — it will be flagged or clobbered on the next build. This is the deliberate, durable point of the change: docs become *owned*, with attribution and drift-guarding, instead of un-owned files.
- **Drift is guarded by two existing `doctor` checks.** `orphaned_publish_targets` (`lib/textus/doctor/check/orphaned_publish_targets.rb`) catches a published file whose source entry was removed; `sentinels` (`lib/textus/doctor/check/sentinels.rb`) catches a managed target that drifted from its sentinel (a hand-edit of generated output). Together they make "someone edited the wrong layer" a detectable, reportable condition rather than silent rot.
- **Zero migration diff, full history.** Adopt-in-place means the landing build touches no bytes in `docs/`; relative links between docs and `git blame` for every line survive intact. A reviewer sees the store fill up and the manifest grow, not a docs reshuffle.
- **No wire change; `SPEC.md` is unaffected.** Publish is repo-local materialization, not a protocol operation — there is no `textus/N` wire field, hook event, or build-envelope field added or changed. **This ADR deliberately does *not* edit `SPEC.md`.** That is the exception worth stating plainly: the `adr` runbook's step 4 normally pairs an ADR with a `SPEC.md` edit (the *what* to this ADR's *why*), and a reader will look for one. There is none, because nothing in the normative protocol changes — this is purely which files the store chooses to own and materialize locally.
- **The `knowledge` zone becomes substantial.** It goes from `project.md` + runbooks to the project's entire committed prose corpus. That is the intended end state of dogfooding (ADR 0041): the store is now genuinely the source of truth for what humans and agents read.
- **The 80 ADRs are owned without being touched.** `schema: null` + adopt-in-place means every historical ADR is now a tracked canon entry while remaining byte-for-byte its committed self — no frontmatter, no reformat, no validation gate retrofitted onto frozen records.

## Alternatives considered

- **Move-and-drop: relocate `docs/` *into* the store and stop publishing a copy back.** Rejected. It breaks every relative link between docs (and from code/READMEs into `docs/`), and it severs `git blame` at the move commit for the most-read prose in the repo. It also changes the on-GitHub reading experience: contributors expect `docs/` to be where docs are. Publishing back out keeps the consumer surface exactly where it is while moving *ownership* into the store — the whole reason ADR 0050's adoption branch exists.
- **Schema-per-doc / add frontmatter to the ADRs.** Rejected. Giving docs a real schema sounds tidier but forces frontmatter onto 80 frozen files (an 80-file churn commit with no behavioral payoff) and inserts a load-time validation gate in front of historical records — risking that a single malformed legacy header makes the store fail to load. Docs are prose; `schema: null` is the honest type for them, and it is what makes zero-diff adoption possible at all.
- **Leave `docs/` flat (status quo / do nothing).** The honest do-nothing option. Rejected because it is exactly the gap that motivates this ADR: `docs/` stays a pile of un-owned files that any writer can silently edit, move, or let rot, with no attribution and no drift guard — in a tool whose entire reason for existing is that durable repo knowledge should be owned, not left loose. A repo-memory tool that exempts its own most-read docs from ownership is not credible.
