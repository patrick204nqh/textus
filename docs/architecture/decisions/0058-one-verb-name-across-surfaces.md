# ADR 0058 — One verb name across surfaces: align the MCP tool, the CLI command, and the use-case method

<!-- Drafting note: an earlier draft of this ADR also renamed the CLI `build` command to `publish` to match the `Write::Publish` use-case. Backed out during execution: `build` is a *capability* (`propose/author/keep/fetch/build`), the `derived` zone-kind's required verb, and pervasive concept-prose in SPEC.md — four layers say "build", only the use-case class says "publish". The right fix is the reverse direction (rename the verb/class to `build`), which collides with the separate manifest `publish:` block (ADR 0052) and is its own decision → roadmap row 0061. -->


**Date:** 2026-06-02
**Status:** Accepted (ships 0.44.0)
**Refines:** [ADR 0036](./0036-transports-as-pure-framings.md) (transports are pure framings over one verb vocabulary — this ADR enforces that the *name* of a verb is also one, not one-per-transport), [ADR 0039](./0039-mcp-catalog-derive-or-guard.md) (the MCP tool name is the contract verb, which is the `Dispatcher::VERBS` key, which is the `RoleScope` method — so a name is changed in exactly one place and the three move together).
**Touches:** [ADR 0057](./0057-agent-legible-mcp-contracts.md) (made each MCP *argument* legible; this makes each verb *name* consistent across the transports an operator and an agent each read), [ADR 0056](./0056-boot-quickstart-speaks-the-mcp-catalog.md) (`boot.read_verbs` is derived from the catalog, so renaming a read verb changes what `boot` advertises — the guard spec moves with it).

> **One sentence:** a senior-architect review of the MCP and CLI surfaces found the architecture clean (one contract, derived everywhere) but several verbs wearing *two different names* — `schema` on MCP vs `schema show` on the CLI, a CLI `fetch stale` that actually calls `fetch_all`, a `retention_sweep` verb whose only command is `retain`, and a redundant top-level `delete` duplicating `key delete` — so this ADR makes the name singular: the MCP tool, the CLI command path, and the use-case method are the same token (groups flatten with `_`).

## Context

textus already has one verb vocabulary lifted to core (ADR 0036) and one declared contract per verb that every transport projects from (ADR 0039). The `Dispatcher::VERBS` key, the `Contract` `verb`, and the `RoleScope`/`Store` method are the *same symbol* by construction — `RoleScope` metaprograms a method per `VERBS` key, and `MCP::Catalog` uses `contract.verb` as both the `tools/list` name and the method it `public_send`s. That is the strong foundation; the gaps are at the naming layer an operator (CLI) and an agent (MCP) actually read, and a reviewer comparing the two surfaces sees the same operation under two names:

1. **`schema` (MCP) vs `schema show` (CLI).** Both reach the *same* use-case (`Read::SchemaEnvelope`); the CLI nests it under the `schema` group (which also holds `diff`/`init`/`migrate`) while MCP exposes the bare token `schema`. The names disagree for one operation.
2. **`fetch stale` (CLI) calls `fetch_all`.** The CLI leaf is named `stale`, but `Verb::FetchStale#call` invokes `session.fetch_all(...)` — it refreshes *all* matched entries, not only stale ones. The MCP tool for the same use-case is correctly named `fetch_all`. The CLI label is actively misleading.
3. **`retention_sweep` (verb) vs `retain` (CLI).** The only command for `Write::RetentionSweep` is `retain`; the verb/method carries the longer name. One operation, two tokens.
4. **Top-level `delete` duplicates `key delete`.** Two CLI commands both resolve to `Write::Delete`: top-level `delete KEY --if-etag=E` and `key delete KEY` (which also handles `--prefix` → `Maintenance::KeyDeletePrefix`). They are not even identical — only the top-level one accepts `--if-etag`, and only `key delete` accepts `--prefix` — so the duplication also *splits* the single-delete capability across two commands.

Three further findings are larger than a rename and are **out of scope here** (pre-registered as roadmap rows, see README):

- **The `rule` family overlaps three ways.** MCP `rules` (`Read::Rules`) returns effective `{fetch, guard}` for one key and has *no* CLI surface; CLI `rule explain KEY` (`Read::PolicyExplain`) is a richer for-key view; CLI `rule list` reads `manifest.rules.blocks` inline with no use-case. `rules` is a thin subset of `policy_explain` under a different name on a different transport. Consolidating the for-key rule introspection is a design decision, not a rename → **row 0059**.
- **The agent (MCP) capability surface is narrower than the operator's.** Single-key `delete`/`mv` are CLI-only while their *bulk* `*_prefix` cousins are on MCP (an agent can delete a whole prefix but not one key); and `where`/`deps`/`rdeps` graph-navigation reads are CLI-only. Surfacing these to MCP needs new contracts → **row 0060**.
- **`build` (CLI) dispatches the `publish` verb, but everything else says "build".** `Verb::Build#call` runs `ops.publish` (`Write::Publish`), yet `build` is also a *capability*, the `derived` zone-kind's required verb, and pervasive concept-prose. The one-name principle here points the *other* direction — rename the verb/class to `build` — which collides with the separate manifest `publish:` block (ADR 0052). That is a vocabulary decision with its own blast radius → **row 0061**.

## Decision

A verb has **one name**. The MCP tool name, the CLI command *path*, and the use-case method are the same token; when the CLI nests a verb under a group, the MCP/method token is the group and leaf joined with `_` (`schema show` → `schema_show`, matching the existing `zone mv` → `zone_mv`, `rule lint` → `rule_lint`, `key mv-prefix` → `key_mv_prefix`).

1. **`schema` → `schema_show`.** Rename the `Dispatcher::VERBS` key, the `Read::SchemaEnvelope` contract `verb`, and therefore the `RoleScope#schema_show` method; the CLI `schema show` leaf and `Verb::Schema` now call `session.schema_show`. The MCP tool is `schema_show`; `boot.read_verbs` advertises `schema_show`. (The manifest entry attribute `entry.schema` and the JSON `Schemas` store are unrelated and unchanged.)
2. **CLI `fetch stale` → `fetch all`.** `Verb::FetchStale` becomes `Verb::FetchAll` with `command_name "all"`; `Group::Fetch` routes `all` (not `stale`) to it. It still calls `fetch_all`; the label now tells the truth. The MCP tool stays `fetch_all`.
3. **`retention_sweep` → `retain`.** Rename the `Dispatcher::VERBS` key so the method is `RoleScope#retain`; the CLI `retain` command calls `session.retain`. (The use-case has no contract — it is CLI-only — so there is no MCP tool name to move; `Read::Retainable`/`retainable`, the preview side, is unchanged.)
4. **Collapse top-level `delete` into `key delete`.** Remove `Verb::Delete` (top-level). Add `--if-etag` to `Verb::KeyDelete` so the single-key path supports optimistic concurrency; `key delete KEY [--if-etag=E]` is the one single-delete command and `key delete --prefix P` remains the bulk one. Drop the `delete` row from `CURATED_CLI_VERBS` (it lives under `key`).
5. **Guard it (ADR 0037/0039).** The derive-or-guard specs that pin the catalog names and `boot.read_verbs` move with the renames (`schema` → `schema_show`); the MCP↔dispatcher reconciliation spec drops `retention_sweep` for `retain`. `SPEC.md` and `docs/reference/mcp.md` reflect the renamed read-verb list.

## Consequences

- **A reviewer comparing the two surfaces sees one name per operation.** The thing an agent calls `schema_show` is the thing a human types as `schema show`; `fetch all` no longer claims to be about staleness; the `retain` command runs the `retain` verb.
- **Breaking changes, no shims (deliberate).** The MCP tool `schema` is renamed to `schema_show`; the CLI commands `fetch stale` and top-level `delete` are removed in favour of `fetch all` and `key delete`. Any client or script hardcoding the old names must update. This is consistent with ADR 0057's stance (the `meta` → `_meta` rename) — the protocol is pre-1.0 and a wrong name is a defect, not a contract to preserve.
- **`boot.read_verbs` output changes** (`schema` → `schema_show`); since it is derived (ADR 0056) the change is automatic and guarded, not hand-mirrored.
- **No use-case logic changes.** Every rename is a token change across the co-moving sites plus call sites; behaviour, args, and responses are identical.
- **The three deeper issues are recorded, not silently dropped.** The `rule`-family overlap (row 0059), the agent capability gap (row 0060), and the `build`/`publish` vocabulary tension (row 0061) are pre-registered so the review's findings are not lost.

## Alternatives considered

- **Rename the CLI to the MCP token instead (bare `schema`, `fetch_all`, etc.).** Rejected: the CLI groups (`schema`, `fetch`, `key`, `rule`) are good operator ergonomics — `schema {show,diff,init,migrate}` reads well. The fix is to make the *MCP/method* token equal the flattened path, not to flatten the CLI.
- **Keep `retention_sweep` as-is** since it is CLI-only and so not strictly an "MCP vs CLI" inconsistency. Rejected: the principle is one name *end to end* (command = verb = method); a command that dispatches a differently-named verb is the same legibility defect one layer down, and cheap to fix when (as here) the rename is unambiguous.
- **Also rename CLI `build` → `publish` in this pass.** Attempted, then rejected: unlike `retention_sweep`, `build` is not a lone internal token — it is a capability, the `derived` zone-kind's required verb, and pervasive concept-prose, all of which say "build". The one-name principle therefore points the *opposite* way (rename the `Write::Publish` verb to `build`), which collides with the manifest `publish:` block (ADR 0052). Reconciling that is a vocabulary decision with its own blast radius → row 0061, not folded in here.
- **Fold the `rule`-family consolidation and the MCP capability gap into this ADR.** Rejected *as part of this ADR* for scope and verifiability: those add/merge use-cases and contracts (and change what an agent can *do*, not just what things are *called*), each with its own blast radius. They were written as separate verifiable decisions — [ADR 0059](./0059-one-rule-verb-two-depths.md) and [ADR 0060](./0060-agent-safety-graph-reads-and-default-dry-run.md), each with its own ADR and specs — and then shipped alongside this one in the 0.44.0 release. (Same release, three distinct decisions; not one folded ADR.)
- **Provide deprecated aliases for the old names.** Rejected, matching house style: textus is pre-1.0, the derive-or-guard machinery makes a name singular by construction, and carrying alias tables would reintroduce the very many-names-for-one-verb state this ADR removes.
